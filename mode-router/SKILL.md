# 工作模式路由 — 分析 vs 修复

## 依赖要求

- 无外部依赖
- 无平台限制
- 无 API Key 要求

---

## 执行步骤

### 步骤 1：扫描用户输入中的模式关键词

从用户输入中搜索以下关键词（优先匹配句末）：

**分析模式关键词**：`分析`、`排查`、`什么原因`、`诊断`、`为什么`、`怎么回事`、`帮忙看看`、`解释一下`、`帮我看看`

**修复模式关键词**：`修复`、`修改`、`实现`、`改`、`做`、`解决`、`帮我把`

### 步骤 2：解析歧义

按以下规则确定模式：

| 场景 | 规则 |
|------|------|
| 只匹配分析关键词 | → 分析模式 |
| 只匹配修复关键词 | → 修复模式 |
| 同时匹配两种 | → 以最靠近句末的明确意图词为准 |
| 两种都不匹配 | → **默认分析模式** |
| 用户说「帮我看看这个 Bug」 | → 分析模式（定位后再问是否修） |
| 用户说「这个怎么改」 | → 分析模式（问方案，未授权改） |
| 两种模式均无法确定 | → 追问：「请问是需要先分析原因，还是直接修复？」 |

### 步骤 3：分析模式 — 只输出方案，不写代码

**输出结构（必须严格遵循）：**

```
## 问题原因
（对问题根因的客观分析，结合项目上下文）

## 修改方案
（至少两种备选方案）

### 方案一：[名称]
- **优点**：...
- **缺点**：...
- **推荐理由**：结合架构一致性、维护成本、实现复杂度说明

### 方案二：[名称]
- **优点**：...
- **缺点**：...
- **推荐理由**：...
```

**强制约束**：
- 禁止输出任何代码修改（包括 diff、代码块、行内修改）
- 禁止使用「修复」「修改」「改为」等执行性动词
- 方案数量 ≥ 2

### 步骤 4：修复模式 — 先分析再确认再改

**流程**：
1. **先分析**：完整执行步骤 3（问题原因 + ≥2 种方案）
2. **后确认**：输出分析后追加确认语：「我将按方案一执行修改，请确认」
3. **等授权**：用户确认后（或用户一开始就明确说「直接改」），才输出代码
4. **改代码**：输出修改后的完整代码
5. **验证**：运行编译/测试，附验证结果

**如果用户在修复模式中给了直接执行许可 → 跳过确认步骤，但分析步骤不可跳过。**

### 步骤 5：自我检查

改完后逐项确认：
- [ ] 分析模式输出不含代码修改
- [ ] 分析模式方案 ≥ 2
- [ ] 修复模式先输出了完整分析
- [ ] 修复模式确认后才输出代码
- [ ] 修改后已执行验证
- [ ] 未在分析模式使用「修复」「修改」「改为」

## 失败处理

| 失败场景 | 表现 | 处理 |
|---------|------|------|
| 关键词误判 | `为什么改这个` 含「为什么」但句末「改」是修复意图 | 以句末意图词为准 → 修复模式 |
| 完全无关键词 | 用户问「这个怎么优化」 | 默认分析模式。输出方案后用户可自行选择是否要改 |
| 用户在分析模式不满意 | 看完方案后说「直接帮我改」 | 切换到修复模式，从步骤 4 开始（已有分析结果，直接确认执行） |
| 用户在修复模式被打断 | 确认后用户追加了新条件 | 回到步骤 3，重新分析新条件下的方案 |
| 验证失败 | 修改后编译/测试不通过 | 报告失败原因，回到步骤 3 重新分析。不掩盖错误 |
| 方案无法二选一 | 两个方案各有不可接受的缺点 | 诚实告知，追问用户看重的优先级，不强行推荐 |

---

## 输出格式

**分析模式**：
```
## 问题原因
...

## 修改方案
### 方案一：...
### 方案二：...
```

**修复模式（确认前）**：
```
（完整分析内容）

---
我将按方案一执行修改，请确认。
```

**修复模式（确认后）**：
```
（代码修改内容）

---
验证结果：编译通过 / 测试通过（N/N passed）
```

---

## 禁止行为

- × 分析模式直接贴代码（包括 diff、代码块）
- × 修复模式跳过分析直接改
- × 只给一种方案
- × 改完不验证声称完成
- × 在分析模式使用「修复」「修改」「改为」等动词
