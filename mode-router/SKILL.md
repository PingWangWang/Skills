# 工作模式路由 — 分析 vs 修复

**Why:**
未区分「分析」和「修复」的输出是代码审查中最常见的噪音源之一。在分析阶段输出代码修改，会让用户被迫从修改片段中反推原因；在修复阶段跳过分析，则导致盲目修改、引入回归。强制二选一的模式路由确保每一步输出都匹配用户真实需求。

**Scope:**
适用于所有涉及代码问题诊断、Bug 修复、功能实现、重构或设计决策的会话。不适用于纯文档编写、配置修改、项目无关问题。

---

## Trigger conditions（满足任一即触发）

1. 用户提出涉及代码、Bug、功能、设计或项目相关的问题。
2. 用户明确使用「分析」「排查」「什么原因」「诊断」「为什么」「怎么回事」「帮忙看看」「解释一下」等分析型关键词。
3. 用户明确使用「修复」「修改」「实现」「改」「做」「解决」「帮我把」等执行型关键词。
4. 用户描述了一个问题现象但未指定要分析还是要修复（歧义场景）。

---

## How to apply

### 模式一：分析问题模式（Analysis Mode）

**触发关键词：** `分析`、`排查`、`什么原因`、`诊断`、`为什么`、`怎么回事`、`帮忙看看`、`解释一下`

**行为约束（强制）：**
- 只输出问题原因分析和修改方案。
- 禁止输出任何具体的代码修改（包括代码块、diff、行内修改）。
- 禁止使用「修复」「修改」「改为」等执行性动词描述当前操作。

**输出结构：**

```
> ## 问题原因
> （对问题根因的客观分析，结合项目上下文）
>
> ## 修改方案
> （从结构、设计原理、编码规范等角度综合考量，至少两种备选方案）
>
> ### 方案一：[名称]
> - **优点**：...
> - **缺点**：...
> - **推荐理由**：结合架构一致性、维护成本、实现复杂度说明
>
> ### 方案二：[名称]
> - **优点**：...
> - **缺点**：...
> - **推荐理由**：...
```

### 模式二：修复问题模式（Fix Mode）

**触发关键词：** `修复`、`修改`、`实现`、`改`、`做`、`解决`、`帮我把`

**行为约束（强制）：**

1. **先分析**：必须完整执行分析模式的全部步骤（问题原因 + 至少两种方案）。
2. **后确认**：输出分析方案后，向用户简要确认：「我将按方案一执行修改，请确认」。
3. **用户授权后**方可输出修改后的代码（或用户已明确表示可直接执行时）。
4. **验证**：代码修改完成后，按项目编译/测试流程执行验证，确保无语法错误。

---

## 歧义处理规则（防冲突）

| 场景 | 规则 |
|------|------|
| 同时含分析+修复关键词 | 以最靠近句末的明确意图词为准 |
| 完全无关键词 | **默认分析模式**（只输出方案，不修改代码） |
| 两种模式均无法匹配 | 追问：「请问是需要先分析原因，还是直接修复？」 |
| 用户说「帮我看看这个 Bug」 | 视为分析模式（定位问题后再问是否要修） |
| 用户说「这个怎么改」 | 视为分析模式（问方案，未授权改） |

## 快速参考

| 用户输入 | 匹配模式 | 行为 |
|----------|----------|------|
| `分析一下这个分段函数的正确性` | 分析 | 原因分析 + 方案（无代码） |
| `为什么这里会段错误` | 分析 | 原因分析 + 方案（无代码） |
| `修复这个内存泄漏` | 修复 | 先分析原因 → 确认 → 改代码 → 编译验证 |
| `帮我把登录改成 OAuth` | 修复 | 先设计方案 → 确认 → 改代码 → 验证 |
| `这个数组越界怎么回事` | 分析 | 原因分析 + 方案（无代码） |
| `解决一下性能瓶颈` | 歧义 → 分析 | 迫问或默认分析模式 |
| `帮我看看这段代码` | 分析 | 审查 + 建议（无直接修改） |

---

## 常见错误

| 错误 | 表现 | 修正 |
|------|------|------|
| 分析模式出代码 | 还没问用户就直接贴修改 | 分析阶段只输出方案描述，不碰代码 |
| 修复模式跳分析 | 用户说「修 Bug」就直接改 | 必须先分析原因 + 方案，确认后再改 |
| 关键词误判 | `为什么改这个` 含「为什么」但意图是修 | 以句末意图词为准，此处「改」为修复模式 |
| 默认选错 | 用户问「这个怎么优化」— 可分析可执行 | 追问问清楚，不默认 |
| 方案单一 | 只给一种方案让用户没得选 | 至少两种，各列优缺点 |
| 改后不验证 | 贴完修改就说「完成」 | 必须运行编译/测试验证 |

## Verification（自我检查清单）

- [ ] 分析模式下输出不含任何代码修改片段
- [ ] 分析模式下方案≥2种，各列优缺点
- [ ] 修复模式下先输出完整的分析内容
- [ ] 修复模式修改前已向用户确认
- [ ] 修改后已执行编译/测试验证
- [ ] 歧义场景已按规则兜底或追问
- [ ] 未在分析模式使用「修复」「修改」「改为」等动词

---

## Output rules

1. **分析模式**：只输出问题分析 + 方案描述，不输出可执行代码。
2. **修复模式**：先分析 → 再确认 → 再输出修改 → 最后验证结果。
3. **方案数量**：至少 2 种，不可只给单一方案。
4. **验证声明**：修改完成后必须附带验证结果（通过或失败）。
5. **禁止行为**：
   - × 分析模式直接贴代码
   - × 修复模式跳过分析直接修
   - × 修完不验证就声称完成
   - × 单一方案无备选
