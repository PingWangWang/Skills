# 问题分析 — 只出方案，不动代码

## 依赖要求

- 无外部依赖
- 无平台限制
- 无 API Key 要求

---

## 执行步骤

### 步骤 1：识别触发场景

以下任一条件满足 → 进入分析模式：
- 用户输入含分析关键词：`分析`、`排查`、`什么原因`、`诊断`、`为什么`、`怎么回事`、`帮忙看看`、`解释一下`、`帮我看看`
- 用户只描述了问题现象，未明确说「改」「修复」「实现」
- 用户意图模糊，无法判断是要分析还是要动手改

**不进入分析模式的场景**：
- 用户明确使用修复关键词：`修复`、`修改`、`实现`、`改`、`做`、`解决`、`帮我把` → 路由到 `fix` 技能
- 纯文档编写、配置修改、闲聊 → 不适用本技能

### 步骤 2：歧义消解

| 场景 | 规则 |
|------|------|
| 同时含分析和修复关键词 | 以最靠近句末的明确意图词为准 |
| 完全无关键词 | **默认分析模式** |
| 「帮我看看这个 Bug」 | 分析模式（定位后再问是否修） |
| 「这个怎么改」 | 分析模式（问方案，未授权改） |
| 仍无法确定 | 追问：「请问是需要先分析原因，还是直接修复？」 |

### 步骤 3：定位根因

结合项目上下文（代码结构、调用链、配置），客观分析问题原因。若信息不足，先追问关键细节。

### 步骤 4：输出分析报告

**输出结构（必须严格遵循）：**

```
## 问题原因
（对问题根因的客观分析，结合项目上下文）

## 修改方案
（至少两种备选方案）

### 方案一：[名称]
- **优点**：...
- **缺点**：...
- **推荐理由**：结合架构一致性、维护成本、实现复杂度说明

### 方案二：[名称]
- **优点**：...
- **缺点**：...
- **推荐理由**：...
```

**强制约束**：
- 禁止输出任何代码修改（包括 diff、代码块、行内代码修改）
- 禁止使用「修复」「修改」「改为」等执行性动词
- 方案数量 ≥ 2
- 若两个方案各有不可接受的缺点 → 诚实告知，追问优先级，不强行推荐

### 步骤 5：自我检查

- [ ] 输出不含任何代码修改（含 diff、代码块）
- [ ] 方案 ≥ 2
- [ ] 未使用「修复」「修改」「改为」等执行性动词
- [ ] 每个方案列出了优点、缺点、推荐理由

---

## 失败处理

| 失败场景 | 处理 |
|---------|------|
| 信息不足无法分析 | 先追问关键细节，不强行输出分析 |
| 完全无法定位根因 | 诚实告知，列出已排除的可能性，建议深入排查方向 |
| 方案无法二选一 | 追问用户看重的优先级，不强行推荐 |
| 用户看完方案不满意 | 追问具体不满意的点，重新分析 |
| 用户看完方案说「直接帮我改」 | 提示用户调用 `fix` 技能，将当前分析结果带入 |

---

## 输出格式

```
## 问题原因
...

## 修改方案
### 方案一：...
### 方案二：...
```

---

## 禁止行为

- × 输出任何代码（包括 diff、代码块、行内建议修改）
- × 只给一种方案
- × 使用「修复」「修改」「改为」等执行性动词
- × 在信息不足时强行输出分析
- × 跳过根因直接给方案
