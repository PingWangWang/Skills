# 分析模式 — 只分析，不动代码

## 依赖要求

- 无外部依赖
- 无平台限制
- 无 API Key 要求

---

## 执行步骤

### 步骤 1：识别触发场景

以下任一条件满足 → 进入分析模式：
- 用户输入含分析/解释类关键词：`分析`、`排查`、`什么原因`、`诊断`、`为什么`、`怎么回事`、`帮忙看看`、`解释一下`、`帮我看看`、`原理`、`机制`、`怎么实现的`、`如何工作`、`对比`、`方案`、`评估`
- 用户只描述了问题现象，未明确说「改」「修复」「实现」
- 用户询问技术原理、设计思路、代码实现细节
- 用户要求对多种方案进行评估、对比或分级推荐
- 用户意图模糊，无法判断是要分析还是要动手改
- **默认路由**：当没有其他技能明确匹配时，默认进入分析模式

**不进入分析模式的场景**：
- 用户明确使用修复/实现关键词：`修复`、`修改`、`实现`、`改`、`做`、`解决`、`帮我把` → 路由到 `fix`/`writing` 技能
- 纯文档编写、配置修改、闲聊 → 不适用本技能

### 步骤 2：歧义消解

| 场景 | 规则 |
|------|------|
| 同时含分析和修复关键词 | 以最靠近句末的明确意图词为准 |
| 完全无关键词 | **默认分析模式** |
| 「帮我看看这个 Bug」 | 分析模式（定位后再问是否修） |
| 「这个怎么改」 | 分析模式（问方案，未授权改） |
| 仍无法确定 | 追问：「请问是需要先分析原因，还是直接修复？」 |

### 步骤 3：确定分析类型

根据用户输入，判断本次分析的具体类型，不同类型的侧重点和输出结构不同：

| 分析类型 | 用户典型输入 | 分析侧重点 |
|---------|------------|-----------|
| **问题排查** | 为什么报错、哪里出了问题、帮忙看下这个 Bug | 根因分析、复现路径、影响范围 |
| **原理分析** | 解释一下这是什么原理、这个机制是怎么工作的、为什么要这样设计 | 底层机制、设计思想、演进原因、权衡取舍 |
| **代码实现分析** | 这段代码怎么实现的、这个模块的结构是怎样的 | 架构设计、模块职责、调用链、关键逻辑、数据流 |
| **方案对比/分级** | 对比这两个方案、有什么方案、帮我评估一下这个方案 | ≥2 个方案的分级对比、适用场景、优缺点、推荐建议 |
| **其他分析** | 不属于以上任一类型 | 根据用户需求灵活调整分析框架和产出 |

> 若用户需求覆盖多种类型（如先分析原理再给方案），可组合输出。

### 步骤 4：收集信息

- 结合项目上下文（代码结构、调用链、配置）进行客观分析
- 若信息不足以给出有质量的分析，先追问关键细节
- 需要时主动读取相关文件来获取准确信息

### 步骤 5：输出分析报告

**输出结构：根据步骤 3 确定的类型动态适配，不固定格式。**

> 核心原则：**用户需要什么，就输出什么**，不强套模板。

以下为各类型的参考输出框架（可组合、可灵活调整）：

---

#### 问题排查型

```
## 根因分析
（对问题根因的客观分析，结合项目上下文，说明为什么会发生）

## 复现路径
（如何稳定复现该问题）

## 影响范围
（哪些模块/功能受到影响，严重程度）

## 修复方向
（不涉及具体代码，说明从哪方面入手可以解决，给出 ≥2 种方向）
### 方向一：[名称]
- **思路**：...
- **优点**：...
- **缺点/风险**：...

### 方向二：[名称]
- **思路**：...
- **优点**：...
- **缺点/风险**：...
```

#### 原理分析型

```
## 核心概念
（解释涉及的核心技术概念）

## 工作机制
（一步步说明工作原理，可配合流程图/时序描述）

## 为什么这样设计
（设计意图、历史演进、与其他方案的对比）

## 核心要点
（关键细节、常见误区、最佳实践）
```

#### 代码实现分析型

```
## 整体架构
（模块定位、职责边界、所属层次）

## 核心流程
（主调用链、关键路径、数据流转）

## 关键逻辑详解
（逐个分析核心函数/类的职责与实现思路）

## 设计亮点/潜在问题
（值得借鉴的点，以及可能存在的隐患）
```

#### 方案对比/分级型

```
## 需求背景
（分析的前提和约束条件）

## 方案对比
### [方案名称]
- **核心思路**：...
- **适用场景**：...
- **优点**：...
- **缺点/风险**：...
- **实施成本**：[低/中/高]

### [方案名称]
...

## 推荐建议
（根据用户需求给出分级的推荐，如：首选 / 备选 / 不推荐，并说明理由）
```

#### 组合/自定义型

若用户需求覆盖多种类型或以上框架都不适用，自由组合或自定结构输出，确保：
1. 逻辑清晰、层次分明
2. 客观中立，有依据
3. 紧贴用户实际需求

---

**通用约束**：
- 禁止输出任何可直接用于修改的代码（包括 diff、代码块、行内代码修改）
- 分析过程中引用代码时，可用文件路径+行号方式指代，不粘贴代码全文
- 确保分析有深度，不只停留在表面描述
- 若两个方案各有不可接受的缺点 → 诚实告知，追问优先级，不强行推荐

### 步骤 6：自我检查

- [ ] 输出中无可直接用于修改的代码（含 diff、完整代码块）
- [ ] 输出格式适配了用户的实际需求，而非硬套模板
- [ ] 有足够的信息支撑分析结论（否则已追问）
- [ ] 方案类分析给出了 ≥2 个选择
- [ ] 分析客观中立，有项目上下文依据

---

## 失败处理

| 失败场景 | 处理 |
|---------|------|
| 信息不足无法分析 | 先追问关键细节，不强行输出分析 |
| 完全无法定位根因 | 诚实告知，列出已排除的可能性，建议深入排查方向 |
| 方案类无法给出 ≥2 个方案 | 诚实告知，解释为什么只有一个可行方向 |
| 用户看完分析不满意 | 追问具体不满意的点，调整分析方向重新输出 |
| 用户看完说「直接帮我改/实现」 | 提示用户调用 `fix`/`writing` 技能，将当前分析结果带入 |

---

## 输出格式（通用）

```
## [分析标题 — 根据分析类型动态确定]
...

## [子章节 — 按需组织]
...
```

---

## 禁止行为

- × 输出任何可直接用于修改的代码（包括 diff、完整代码块、行内建议修改）
- × 硬套不合适的模板
- × 跳过分析直接给方案
- × 在信息不足时强行输出
- × 只罗列现象不分析根因
