# 项目无关请求识别与上下文切换

**Why:**
项目协作中，用户可能穿插闲聊、通用咨询等与代码库无关的请求。若将这些请求错误纳入项目上下文，会导致误操作（如在非项目模式修改文件）、缓存污染（无关数据混入项目状态）、输出混乱（回答混代码库内容）。需明确分隔机制以规避风险。

**Scope:**
仅适用于已激活项目上下文的会话场景（如代码分析、提交日志生成、功能开发等）。非项目场景默认按通用模式处理。

---

## Trigger conditions（满足任一即触发）

1. **中文前缀匹配**：输入以 `对话`、`聊天`、`闲聊`、`独立要求` 开头。
2. **英文前缀匹配**：输入以 `chat`、`talk`、`independent request` 开头。
3. **显式分隔符**：输入包含 `---` 或 `===`，其后内容无项目特征词。
4. **领域无关问题**：问题与当前项目技术栈无关（如 `Python 安装方法`、`量子力学解释`）。
5. **纯工具性请求**：用户要求执行与项目无关的操作（如算数、翻译、通用写作）。

**例外 — 不触发切换：**
- 触发词出现在文本中间（非开头），且前后文属项目相关。
- 用户刚在项目模式下提到项目术语后紧跟触发词。
- 输入包含项目文件路径或代码引用（即使含前缀词）。

---

## How to apply

### 1. 实时解析

1. 提取输入前 10 字符扫描关键词。
2. 检查分隔符后内容是否无项目特征词（文件路径、函数名、技术栈术语）。
3. 触发后立即终止项目上下文关联流程。

### 2. 上下文切换

1. 清空项目相关缓存（文件状态、代码引用、提交日志等）。
2. 切换至通用模式，仅依赖基础能力与实时联网。
3. **禁止引用**项目路径、代码、配置信息。

### 3. 安全防护

| 场景 | 处理 |
|------|------|
| 含糊表述（如 `这个怎么改`） | 二次确认：「这个问题是否关联当前项目？」 |
| 危险操作（如 `删除所有文件`） | 即使含触发词也追加风险提示 |
| 触发词 + 文件路径混合 | 切换但拒绝操作，警告检测到文件路径 |
| 前后文频繁切换 | 稳定在当前模式，避免来回跳转 |

## 快速参考

| 输入 | 匹配 | 行为 |
|------|------|------|
| `对话 今天天气如何` | 前缀触发 | 切换通用模式，返回天气 |
| `独立要求：TCP 三次握手` | 前缀触发 | 切换通用模式，返回协议说明 |
| `--- 什么是递归` | 分隔符触发 | 切换通用模式，返回解释 |
| `fix 登录bug` | 项目关键词，无前缀 | 不切换，保持项目模式 |
| `对话：修改src/main.cpp` | 前缀 + 路径混合 | 切换但拒绝操作 |
| `这个怎么改` | 含糊，无项目词 | 二次确认后决定 |
| 纯代码问题 | 无触发条件 | 保持项目模式 |

## 常见错误

| 错误 | 表现 | 修正 |
|------|------|------|
| 误判关键词 | `对话` 出现在文本中间而非开头仍触发 | 严格匹配输入开头 |
| 漏判领域相关 | 用户问「这个函数怎么优化」无前缀但不属闲聊 | 添加项目关键词白名单 |
| 切换时泄露 | 通用回答仍引用项目配置 | 清空项目缓存后再响应 |
| 来回跳转 | 项目→闲聊→项目频繁切换 | 稳定策略，短间隔内倾向保持当前模式 |
| 静默失败 | 切换后用户不知情，以为还在项目模式 | 不声明切换，但危险操作需警告 |

## Verification（自我检查清单）

- [ ] 触发词匹配输入开头（非中间）
- [ ] 项目上下文缓存已清空
- [ ] 回答不包含项目路径/代码/配置
- [ ] 脱离项目回答后不混入项目上下文
- [ ] 危险操作已追加提示（如适用）
- [ ] 含糊请求已二次确认（如适用）
- [ ] 未输出「已退出项目上下文」等模式声明
- [ ] 联网回答标注了来源

---

## Output rules

1. **切换静默**：不声明模式变更，直接响应字面需求。
2. **通用回答规范**：
   - 不涉及"当前项目"或专属上下文。
   - 技术问题优先引用最新公共知识。
   - 联网信息标注来源。
3. **二次确认格式**：`「[问题]」是否指当前项目？` — 不含预设结论。
4. **禁止行为**：
   - × 提示「已退出项目上下文」
   - × 在通用回答中混入代码库内容
   - × 输出上下文切换状态的声明
