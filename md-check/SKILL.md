# Markdown 文档规范检查

## 依赖要求

- 无外部依赖
- 无平台限制
- 无 API Key 要求
- 仅需读取目标 Markdown 文件

---

## 执行步骤

### 步骤 1：确认检查目标

询问用户要检查的文件路径，或根据上下文自动确定。支持：
- 单个 Markdown 文件
- 目录下所有 `.md` 文件（排除 `node_modules/`、`.git/`、`vendor/`）

若用户未指定，默认检查当前对话中涉及的 Markdown 文件。

### 步骤 2：读取文档并逐项检查

按以下 8 项规则逐项检查，记录所有不符合项。

---

#### 检查项 2：标题格式规范

**规则**：各级标题必须使用 Markdown 标准格式（`#` + 空格 + 标题文本），禁止使用 HTML 标签或加粗模拟标题。

**检测方法**：
- 检查所有标题是否以 `#` 开头且 `#` 后紧跟一个空格（即 `# 标题`、`## 标题` 等格式）
- 检查是否有使用 `<h1>`–`<h6>` HTML 标签替代 Markdown 标题
- 检查是否有使用 `**粗体文本**` 或 `__粗体文本__` 单独成行模拟标题
- 检查标题层级是否连续（不应从 `#` 直接跳到 `###`，中间应经过 `##`）

**不合规示例**：
```markdown
❌ <h2>概述</h2>                    → 应使用 ## 概述
❌ **1. 背景介绍**                   → 应使用 ### 1. 背景介绍
❌ #标题（缺少空格）                  → 应使用 # 标题
❌ # 概述 → 直接跳到 ### 详细说明     → 层级不连续，中间缺 ##
```

---

#### 检查项 3：序号格式规范

**规则**：各级标题中的序号必须使用全数字格式（如 `1.`、`1.1`、`2.1.1`），禁止使用中文序号、罗马数字、字母序号等非数字序号。

**检测方法**：
- 检查标题中是否包含以下非数字序号模式：
  - 中文序号：`一、` `二、` `三、` `（一）` `（二）`
  - 罗马数字：`I.` `II.` `III.` `IV.`
  - 字母序号：`a)` `b)` `A.` `B.` `(a)` `(b)`
  - 中文数字：`第一` `第二` `第三`
  - 其他序号：`①` `②` `③` `⑴` `⑵`
- 检查标题中的序号格式是否统一（同一层级的序号风格应一致）

**合规示例**：
```markdown
✅ ## 1. 概述
✅ ### 1.1 背景
✅ ### 1.2 目标
✅ ## 2. 设计方案
✅ ### 2.1 架构设计
✅ ### 2.1.1 模块划分
```

**不合规示例**：
```markdown
❌ ## 一、概述              → 应改为 ## 1. 概述
❌ ### （一）背景            → 应改为 ### 1.1 背景
❌ ## I. 引言               → 应改为 ## 1. 引言
❌ ### a) 安装步骤           → 应改为 ### 1.1 安装步骤
```

---

#### 检查项 4：冗余分隔符检测

**规则**：删除无意义的冗余分隔符 `---`（水平线），保留有实际用途的（如 YAML frontmatter 闭合、表格分隔行等）。

**检测方法**：
- 扫描文档中所有 `---`、`***`、`___` 水平分隔线
- 判断是否冗余：
  - YAML frontmatter 的 `---` 包裹 → **保留**（不报告）
  - 表格中的 `|---` 分隔行 → **保留**（不报告）
  - 连续出现 2 条及以上分隔线 → **冗余**（报告）
  - 分隔线前后均无内容（空白行之间）→ **冗余**（报告）
  - 仅用于视觉装饰、无逻辑分隔意义 → **冗余**（报告）

**判断标准**：如果移除该分隔线后文档逻辑结构不受影响，则该分隔线为冗余。

---

#### 检查项 5：文档名称不作为一级标题

**规则**：文档的文件名（不含扩展名）不应作为文档内的一级标题出现。一级标题应从正式内容开始。

**检测方法**：
- 获取文档文件名（不含 `.md` 扩展名）
- 检查文档中第一个 `#` 一级标题的文本是否与文件名相同或高度相似（如大小写变体、下划线/连字符替换为空格）
- 检查一级标题是否属于"文档名/项目名"性质而非内容章节性质

**不合规示例**：
```markdown
文件名为 README.md：
❌ # README              → 不应以文件名作为一级标题

文件名为 api-design.md：
❌ # API Design          → 不应以文件名作为一级标题
```

**合规示例**：
```markdown
文件名为 README.md：
✅ # 项目概述             → 以内容章节作为一级标题

文件名为 api-design.md：
✅ # 1. API 设计规范       → 以内容章节作为一级标题
```

---

#### 检查项 6：Mermaid 图表语法检查

**规则**：文档中的流程图、时序图、类图等图表应统一使用 Mermaid 语法，并确保语法正确。

**检测方法**：

1. **检查图表是否使用 Mermaid**：
   - 扫描文档中所有图表描述
   - 检查是否有使用 ASCII art 绘制流程图（如 `+--+`、`|  |`、`-->` 等字符画）
   - 检查是否有使用纯文本描述流程（如"步骤1 → 步骤2 → 步骤3"）
   - 发现有非 Mermaid 的图表 → 报告并建议转换为 Mermaid

2. **Mermaid 语法检查**：
   - 检查 ` ```mermaid ` 代码块是否缺少图表类型声明（`graph`、`flowchart`、`sequenceDiagram`、`classDiagram`、`stateDiagram`、`erDiagram`、`gantt`、`pie`、`gitGraph` 等）
   - 检查 `flowchart`/`graph` 中的节点 ID 是否定义、箭头语法是否正确
   - 检查 `sequenceDiagram` 中的参与者是否声明、消息箭头是否完整
   - 检查 `subgraph`/`end` 是否正确配对
   - 检查花括号、方括号、圆括号是否成对闭合

**常见 Mermaid 语法错误**：
```
❌ flowchart LR          → 应为 flowchart LR（正确，但节点未定义或箭头不完整时需要检查）
❌ A --> B:              → flowchart 中箭头后不应有冒号
❌ participant A         → 缺少 sequenceDiagram 声明
❌ subgraph 模块A         → 缺少对应的 end
❌ graph TD; A-->B;      → 分号不是标准分隔符（应使用换行）
```

---

#### 检查项 7：只修复检查出的问题，不修改无关内容

**规则**：本技能**只允许修复检查出的问题**（如替换 emoji、修正标题格式、调整序号等），**禁止修改与检查项无关的内容**。所有修复必须精确对应报告中列出的问题点，不得对文档做额外的内容增删改。

**严格约束**：
- 仅对报告中列出的问题执行定向修复，不修改文档中未报告的内容
- 每个修复必须能追溯到对应的检查项和问题行号
- 禁止借修复之名对文档做格式美化、措辞优化、结构调整等与检查项无关的改动
- 修复应使用精确的编辑（如 edit_file、multi_edit），而非整体重写文件

---

#### 检查项 8：标题简洁性与正式度

**规则**：各级标题应简明扼要、用语正式。

**检测方法**：
- 检查标题长度是否超过 40 个字符（中英文混合），超过则报告"标题过长"
- 检查标题是否包含口语化/非正式表达：
  - 口语词：`搞定`、`搞一个`、`看看吧`、`好用吗`、`超简单`、`贼好用`、`别慌`、`来一个`、`整一个`
  - 疑问句标题（规范文档不宜用问句作标题，FAQ 除外）：`为什么需要 X？`、`怎么安装？`
  - 感叹句标题：`太棒了！`、`注意！`
  - 模糊表达：`一些说明`、`其他内容`、`补充一下`、`杂项`
- 检查标题是否过于宽泛或空洞：
  - `概述` → 可接受（一级标题的常见用词）
  - `其他`、`杂项`、`补充`、`附` → 过于模糊，建议更具体

**合规示例**：
```markdown
✅ ## 1. 系统架构设计
✅ ### 2.1.3 数据库连接池配置
✅ ## 3. 部署与运维
```

**不合规示例**：
```markdown
❌ ## 搞定这个配置超简单，来看看吧                    → 过长 + 口语化
❌ ### 为什么要用 Redis？                             → 疑问句标题
❌ ## 其他内容                                        → 过于模糊
```

### 步骤 4：分级汇总

对问题按严重程度分级：

| 级别 | 标识 | 说明 | 示例 |
|------|------|------|------|
| **错误** | 🔴 | 违反核心规范，影响文档可读性 | 标题格式错误、序号不统一、Mermaid 语法错误 |
| **建议** | 🟡 | 优化建议，不影响基本阅读 | 标题过长、口语化用词、可选的 emoji 替换 |
| **提示** | 🔵 | 轻微改进建议 | 分隔符冗余、标题可更精确 |

---

## 失败处理

| 失败场景 | 处理 |
|---------|------|
| 文件不存在 | 输出「错误：文件 `<路径>` 不存在」，终止检查 |
| 文件非 Markdown 格式 | 输出「提示：文件 `<路径>` 非 `.md` 扩展名，仍按 Markdown 解析」 |
| 文件为空 | 输出「提示：文件 `<路径>` 为空，无需检查」 |
| Mermaid 代码块语法不确定 | 标记为「待人工确认」，提供上下文和疑似问题 |
| 超大文件（>5000 行） | 分段检查，先检查前 2000 行，询问是否继续 |

---

## 输出格式

检查结果始终使用上述步骤 3 的报告格式，直接输出至控制台。关键约束：
- 每个问题必须标注所在行号
- 每条问题给出具体修复建议
- 统计总数和通过率
- 不修改与检查项无关的内容
- 报告内容简洁扼要，避免冗长描述

---

## 禁止行为

- × 修改与检查项无关的内容（仅允许对检查出的问题进行定向修复）
- × 借修复之名对文档做格式美化、措辞优化、结构调整等额外改动
- × 跳过问题直接给出「文档没有问题」的结论（必须逐项检查）
- × 对不确定的问题给出确定性的错误/正确判断（应标记为「待人工确认」）
- × 检查非 Markdown 文件时仍套用全部规则（应提示并询问）
- × 在报告中输出完整的文档内容（仅引用相关行和上下文）
