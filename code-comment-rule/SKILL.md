# 代码注释规范

## 依赖要求

- 无外部依赖
- 无平台限制
- 无 API Key 要求
- 仅需知道目标文件的编程语言（自动检测）

---

## 执行步骤

### 步骤 1：判断是否需要写注释

遇到以下场景 → 必须写注释：

| 场景 | 动作 |
|------|------|
| 创建函数/方法/类 | 写 Docstring（三要素：功能、参数、返回值） |
| 写复杂逻辑分支 | 行注释解释 Why |
| 用魔数/常量 | 行注释解释含义和来源 |
| 修改既有代码 | 加 `[修改]` 标记，说明原因和关联编号 |
| 写 TODO | 写清计划和理由 |

不写注释的场景：简单 getter/setter（一行 Docstring 即可）、自描述的变量名、批量格式化/重命名。

### 步骤 2：写 Docstring（函数头）

按以下模板，根据语言选择语法：

```
功能描述（一句话）
参数:
    param1: 类型，含义，约束
    param2: 类型，含义，约束
返回:
    类型，含义
```

语言语法速查：

| 语言 | 语法 |
|------|------|
| Python | `"""..."""` |
| JS/TS | `/** ... */` |
| Java | `/** ... */` |
| Go | `// 函数名 ...` (行注释约定) |
| Rust | `///` |
| C/C++ | `/** ... */` 或 `///` |
| SQL | `--` |
| Shell | `#` |

### 步骤 3：写行注释（必须 Why）

在代码行**上方**或**行尾**写 `#` / `//` 注释。内容必须解释「为什么这么写」，不重述代码。

```
✅ if retry_count > 3:  # 超过重试上限后静默降级，避免级联故障
❌ if retry_count > 3:  # 如果重试次数大于3
```

### 步骤 4：检查修改留痕

修改既有代码时 → 在修改处添加：
```
# [修改] 原因：接口 v2 移除了 address 字段，改用 components 数组
#       见 JIRA-1234
```

### 步骤 5：自查

改完代码后逐项确认：
- [ ] 新函数有 Docstring
- [ ] 注释说 Why 不说 What
- [ ] 无过期注释
- [ ] 修改处有 `[修改]` 标记
- [ ] 无纯重述注释（如 `i += 1  # 递增 i`）
- [ ] 魔数有含义说明

## 失败处理

| 失败场景 | 表现 | 处理 |
|---------|------|------|
| 不确定代码意图 | 看不出来为什么这么写 | 加 `# TODO: 确认此逻辑的原因`，不做猜测 |
| 不确定语言 | 无法判断用哪种注释语法 | 默认 `//` 行注释 |
| 注释和代码冲突 | 注释说 A，代码做 B | 以代码为准，标记注释为 `# [待确认]`，不改代码逻辑 |
| 函数太复杂无法一句话描述 | Docstring 写太长 | 拆函数，每个小函数单独写 Docstring |
| 第三方库代码 | `.vendor/` 或 `node_modules/` | 跳过，不修改 |

---

## 输出格式

**写代码时**：注释直接附在代码中输出，不额外包装。

**审查代码时**：
```
[文件:行号] 问题类型: 具体问题 → 修复建议
```

**禁止输出**：
- × 纯重述注释
- × 英文单词注释（如 `# variable`）
- × 显而易见的注释（如 `# 打印 hello`）

---

## 禁止行为

- × 在分析模式下输出代码修改
- × 修改代码不更新注释
- × 用注释掩盖烂代码（烂代码应重构，不是加注释解释）
- × 注释中使用非中文（术语、标识符、链接除外）
