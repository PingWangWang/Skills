# 代码注释规范

**Why:**
代码注释是团队协作的无声契约。规范注释降低阅读成本、减少误判、加速审查，并在代码转交时保留设计决策上下文。无注释或劣质注释导致的「读不懂-不敢改-改出错」链条，是遗留系统腐烂的主要原因之一。

**Scope:**
所有产出、修改、审查的代码文件。适用于任何编程语言，注释语法随语言调整（`//`、`#`、`/* */`、`--` 等）。

**核心原则：说 Why 而不是 What。** 代码本身已经说明「做了什么」，注释只需说明「为什么这么做」。

---

## Trigger conditions（满足任一即激活）

1. **产出新代码**：创建函数、类、接口时需写 Docstring。
2. **修改既有代码**：调整逻辑、替换算法、修复边界条件。
3. **审查他人代码**：检查注释是否缺失或误导。
4. **提交或 PR 前**：确认注释与代码一致，无过期注释。

---

## How to apply

### 规则 1：函数头强制 Docstring

每个函数/方法/类定义上方，必须包含完整 Docstring，结构如下：

```
三要素：功能描述 | 参数列表（类型+含义） | 返回值（类型+含义）
```

```
# 好的例子
def calculate_interest(principal: float, rate: float, years: int) -> float:
    """计算复利本息和

    参数:
        principal: 本金（正数）
        rate: 年利率（百分比，如 5 表示 5%）
        years: 投资年限

    返回:
        本息和（本金 + 复利），保留两位小数
    """
    ...
```

```
# 坏的例子 — 缺少参数含义
def calc(p, r, y):
    """计算利息"""
    ...
```

**约定**：简单 getter/setter 可用一行 Docstring；复杂逻辑必须多行。

### 规则 2：行级注释必须写 Why

逻辑判断、循环、算法计算、边界条件、非显而易见的代码，必须在代码行**上方**或**行尾**注释解释「为什么这么写」，而不是「这行代码是什么」。

```
# ✅ 好的行注释 — 解释原因
if retry_count > 3:          # 超过重试上限后静默降级，避免级联故障
    return cached_result     # 缓存可能过期，但比返回错误好

# ❌ 坏的行注释 — 只重述代码
if retry_count > 3:          # 如果重试次数大于3
    return cached_result     # 返回缓存结果
```

```
# ✅ 好的逻辑注释 — 解释设计决策
for user in users:
    if user.is_deleted:      # 逻辑删除的用户仍保留在表中以便审计追溯，但跳过业务处理
        continue
    process(user)
```

### 规则 3：修改留痕

修改既有代码时，在修改处添加注释标记，说明修改原因和意图。避免后人面对不明修改无从下手。

```
# [修改] 原因：接口 v2 移除了 address 字段，改用 components 数组
#           见 JIRA-1234
OLD_VALUE = calculate(input)
```

**约定**：任务关联编号（JIRA/GitHub Issue）优于抽象描述。批量格式化/重命名无需标记。

### 规则 4：注释即文档 — 随代码更新

修改代码时必须同步更新相关注释。过期注释比没有注释更危险——它会误导后续维护者。

修改功能前先查找相关注释 → 评估是否需要更新 → 改动代码的同时改动注释。

### 规则 5：禁止无意义注释

禁止输出以下类型注释：

```
# ❌ 纯重述
i += 1  # 递增 i

# ❌ 英文单词注释（除非术语解释）
x = y  # variable

# ❌ 废纸注释
import os  # 导入os模块

# ❌ 显而易见
print("hello")  # 打印hello
```

## 快速参考表

| 场景 | 必须写 | 可省略 | 格式 |
|------|--------|--------|------|
| 函数定义 | Docstring（功能/参数/返回值） | 简单 getter/setter | `"""中文描述"""` 或语言对应语法 |
| 复杂逻辑分支 | Why + 边界说明 | — | `# 原因：...` 行上方 |
| 魔数/常量 | 含义 + 来源 | 显而易见的（如 `MAX_RETRY` 已自描述） | `TIMEOUT = 30  # 上游 SLA 承诺 25s，留 5s 余量` |
| 修复 Bug | 根因 + 影响 | 格式化调整 | `# [修改] 原因：... 见 #1234` |
| 性能优化 | 优化策略 + 为什么此策略有效 | 微优化（如 `+1` vs `++`） | `# 预分配避免触发 rehash` |
| TODO 标记 | 计划 + 原因 | — | `# TODO: 接入统一配置中心后替换此硬编码` |

## 语言差异说明

| 语言 | Docstring 语法 | 行注释 | 块注释 |
|------|---------------|--------|--------|
| Python | `"""三引号"""` | `#` | — |
| JavaScript/TypeScript | `/** JSDoc */` | `//` | `/* */` |
| Java | `/** Javadoc */` | `//` | `/* */` |
| Go | `// 函数名...` (无原生 docstring 语法，约定 //) | `//` | — |
| Rust | `///` 或 `//!` | `//` | `/* */` |
| C/C++ | `/** */` 或 `///` | `//` | `/* */` |
| SQL | — | `--` | `/* */` |
| Shell | — | `#` | — |

## 常见错误

| 错误 | 表现 | 修正 |
|------|------|------|
| 写 What 不写 Why | `// 循环遍历数组` | 删除或改写为 `// 按创建时间降序排列，最新的在前` |
| 注释过期 | 注释说 A，代码做 B | 修改代码同时更新注释 |
| 注释太泛 | `# 处理异常` | 具体说明什么异常、为什么、怎么处理 |
| 过度注释 | 每行都写，噪音掩盖信号 | 仅注释非显而易见处 |
| 写死术语不解释 | `// QPS 阈值熔断` | 补充：`// 单机 QPS 超过 500 触发熔断，预估峰值 400 留 20% 余量` |

## Verification（自我检查清单）

- [ ] 每个函数/方法有 Docstring（三要素齐全）
- [ ] `//` 或 `#` 开头的注释解释「为什么」而非「是什么」
- [ ] 无过期注释（注释内容与代码逻辑一致）
- [ ] 修改处有 `[修改]` 标记（如适用）
- [ ] 无纯英文单词注释 / 无意义重述
- [ ] 魔数字面量有含义说明
- [ ] TODO 有计划和理由（不只有 TODO 二字）

---

## 输出规则

1. **语言**：注释全部使用中文（术语、代码标识符、外部链接等原文保留）。
2. **格式**：遵循语言社区约定（Python 用 `"""`，JS/TS 用 `/** */`，Rust 用 `///`）。
3. **位置**：函数头在定义上方，行注释在相关代码行上方或行尾。
4. **完整性**：修改代码必须同时更新注释，禁止只改代码不改注释。
5. **拦截**：审查时发现「纯重述注释」必须标记并要求删除。
