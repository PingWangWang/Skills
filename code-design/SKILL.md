# 方案设计 — 需求 → 调研 → 设计 → 选型

## 依赖要求

- 无外部依赖
- 无平台限制
- 无 API Key 要求
- 需要读取项目源码进行调研

---

## 执行步骤

### 步骤 1：识别触发场景

以下任一条件满足 → 进入设计模式：
- 用户提新功能需求（"新增"、"设计"、"开发"、"做个"）
- 用户问架构选型（"用什么方案"、"怎么设计"、"如何实现"）
- 用户要求技术方案评审
- 用户说"设计方案"、"画架构"、"技术设计"

**不进入设计模式的场景**：
- Bug 修复 → 路由到 `code-analysis`
- 已确定方案要求直接写代码 → 路由到 `code-writting`
- 纯代码审查 → 路由到 `code-revirwer`

### 步骤 2：理解需求

向用户澄清以下事项（用户已明确说清的可跳过询问）：

| 事项 | 问什么 | 示例 |
|------|--------|------|
| 业务目标 | 要解决什么核心问题？ | 用户管理页面支持角色权限 |
| 约束条件 | 技术栈、性能、工期限制？ | 必须用 Vue3，首屏 < 1s |
| 边界范围 | 这次做哪些、不做哪些？ | 只做查询不做编辑 |
| 用户画像 | 谁在用？什么规模？ | 内部管理员，50 人 |
| 验收标准 | 怎么算做好？ | 权限配置后 5s 内生效 |

对模糊的需求，追问不超过 3 轮。超过 3 轮仍不明 → 基于最合理的假设出方案，并在方案中标注假设点。

### 步骤 3：熟悉项目

根据需求范围读取项目关键文件：

```bash
# 查看项目结构
ls <project-root>/src

# 查看现有路由/页面组织
find <project-root> -type f -name "*.ts" | head -20

# 查看现有数据流/状态管理
# 查看现有 API 层
```

必须读的内容：
- **项目结构**：目录组织、模块划分
- **现有数据流**：状态管理、API 层、数据模型
- **相关功能模块**：与需求相近的已有实现
- **技术栈配置**：package.json、tsconfig、构建配置

不需读的：与需求无关的文件、第三方库源码。

### 步骤 4：分析关键设计维度

从以下维度评估（按需求选择相关维度，不必全部覆盖）：

| 维度 | 关注点 |
|------|--------|
| 架构 | 分层、模块解耦、可扩展性 |
| 数据流 | 单向/双向、状态管理、缓存策略 |
| 组件设计 | 复用性、职责划分、props 粒度 |
| 接口设计 | RESTful/gRPC、参数校验、错误处理 |
| 路由设计 | 嵌套路由、权限守卫、懒加载 |
| 安全性 | 鉴权、防越权、数据脱敏 |
| 性能 | 渲染策略、请求合并、懒加载 |
| 兼容性 | 现有模块侵入程度、迁移成本 |

### 步骤 5：设计 ≥2 个方案

每个方案必须包含：

```
## 方案 X：<方案名称>

**一句话概括**：...

**核心思路**：...

**详细设计**：
1. 模块/组件划分
2. 数据流设计
3. 关键接口定义
4. 变更文件列表

**优点**：...
**缺点**：...
**适用场景**：...
```

**方案设计原则**：
- 方案之间必须有本质差异（不只是实现方式不同）
- 至少覆盖两种不同思路（如"最小改动" vs "最佳架构"）
- 如果有推荐方案，放在方案一

### 步骤 6：输出方案对比

```
## 方案对比

| 维度 | 方案一 | 方案二 | 方案三（如有） |
|------|--------|--------|---------------|
| 工作量 | ? 人天 | ? 人天 | ? 人天 |
| 侵入性 | 高/中/低 | 高/中/低 | 高/中/低 |
| 可扩展性 | 高/中/低 | 高/中/低 | 高/中/低 |
| 风险 | ... | ... | ... |

## 建议

推荐方案 X，理由：...
```

### 步骤 7：自我检查

- [ ] 需求理解已确认（或已标注假设）
- [ ] 已阅读相关项目代码，了解现有实现
- [ ] 输出了 ≥2 个有本质差异的方案
- [ ] 每个方案都有明确的优缺点分析
- [ ] 有方案对比表和推荐
- [ ] 没有输出任何具体代码实现

## 失败处理

| 失败场景 | 处理 |
|---------|------|
| 需求信息不足 | 追问，不超过 3 轮；仍不足则标注假设出方案 |
| 项目结构复杂找不到关键文件 | 输出当前目录结构，让用户指明关注模块 |
| 技术栈不熟悉 | 基于常规最佳实践出方案，标注"按 XX 技术栈通用做法" |
| 方案不止两种 | 选最有差异性的 2-3 个方案，其余放备注 |
| 用户临时追加需求 | 回到步骤 2 重新理解需求范围 |
| 方案之间存在优劣明显 | 推荐方案放第一，说明推荐理由，其他方案列出但标注"不推荐" |

## 禁止行为

- × 跳过需求理解直接出方案
- × 不读现有代码就设计
- × 只给一种方案
- × 方案之间没有本质差异（换 API 名称不算新方案）
- × 在设计阶段输出具体代码实现
- × 使用模糊描述（"优化"、"改进"、"提升质量"）
- × 设计方案超出需求范围
- × 忽略现有代码风格和约定
- × 对不清楚的假设不做标注
