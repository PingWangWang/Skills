# 问题修复 — 分析 → 修改 → 注释 → 编码 → 验证

## 关联技能（本技能执行过程中必须同步调用）

本技能是修复流程的编排器，执行过程中必须串联以下三个技能：

| 技能 | 调用时机 | 作用 |
|------|---------|------|
| `code-analysis` | 步骤 3（修改前） | 分析根因，提出 ≥2 种方案 |
| `code-comment` | 步骤 5（修改时） | 确保新增/修改的代码符合注释规范 |
| `file-encoding` | 步骤 5（修改后） | 确保修改过的文件编码为 UTF-8 with BOM + LF |

**调用规则**：
- `code-analysis` → 必须执行，不可跳过（除非用户已从 code-analysis 转来且带分析结果）
- `code-comment` → 每次写入/修改代码文件时，同步检查注释合规性
- `file-encoding` → 每次修改文件后，检测并修复编码和换行符

## 执行步骤

### 步骤 1：识别触发场景

以下任一条件满足 → 进入修复模式：
- 用户输入含修复关键词：`修复`、`修改`、`实现`、`改`、`做`、`解决`、`帮我把`
- 用户已从 `code-analysis` 技能获得分析报告，明确要求动手改
- 用户一开始就给出直接执行许可（如「直接帮我改」「不用分析，动手」）

**不进入修复模式的场景**：
- 用户只用了分析关键词（`为什么`、`怎么回事`）→ 路由到 `code-analysis` 技能
- 纯文档编写、闲聊 → 不适用本技能

### 步骤 2：歧义消解

| 场景 | 规则 |
|------|------|
| 同时含分析和修复关键词 | 以最靠近句末的明确意图词为准 |
| 「帮我看看这个 Bug」 | 分析模式，不进入修复 |
| 「直接改」「不用确认」 | 修复模式，跳过确认步骤 |
| 用户从 code-analysis 转来 | 已有分析结果，直接从步骤 4 开始 |

### 步骤 3：先分析 — 调用 code-analysis（不可跳过）

**调用 `code-analysis` 技能**，即使修复意图明确也必须执行。输出格式：

```
## 问题原因
（对问题根因的客观分析）

## 修改方案
（至少两种备选方案，含优缺点和推荐理由）
```

**如果用户从 `code-analysis` 转来且已有分析结果 → 引用已有分析，跳过此步骤。**

### 步骤 4：确认（用户已授权可跳过）

输出分析后追加：

```
---
我将按方案一执行修改，请确认。
```

等待用户确认后再动手。以下情况可跳过确认：
- 用户一开始就说「直接改」「不用分析」「动手」
- 用户从 code-analysis 转来时说了「就按方案一改」
- 用户在当前对话中已给出明确执行许可

### 步骤 5：执行修改 + 同步调用 code-comment 和 file-encoding

用户确认后，执行修改。每修改一个文件，同步执行以下检查：

**5a. 代码注释检查 — 调用 `code-comment`**：
- 新增函数/类 → 必须有符合规范的注释
- 修改的代码块 → 注释需同步更新
- 不满足 → 补齐注释后再继续

**5b. 文件编码检查 — 调用 `file-encoding`**：
- 对每个修改过的源码文件，检测编码、BOM、换行符
- 不合规 → 自动转换为 UTF-8 with BOM + LF
- 跳过第三方目录（`vendor/`、`build/` 等）

**修改原则**：
- 手术式修改：只改必须改的地方
- 遵循现有代码风格
- 标明修改的文件路径

### 步骤 6：验证

修改完成后执行验证：
- 编译检查（如适用）
- 运行相关测试（如适用）
- 人工确认关键逻辑正确

输出验证结果：
```
---
验证结果：编译通过 / 测试通过（N/N passed）
编码检查：N 个文件已确认 UTF-8 with BOM + LF
注释检查：N 个新增/修改函数注释已合规
```

**验证失败 → 报告失败原因，回到步骤 3 重新分析。不掩盖错误。**

### 步骤 7：自我检查

- [ ] 修复前先输出了完整分析（code-analysis）
- [ ] 确认后才输出代码（除非用户已授权跳过）
- [ ] 修改范围不超出问题本身
- [ ] 新增/修改的函数和类有合规注释（code-comment）
- [ ] 修改过的文件编码已检查/修复（file-encoding）
- [ ] 已执行验证并报告结果
- [ ] 验证失败未掩盖

## 失败处理

| 失败场景 | 处理 |
|---------|------|
| 信息不足无法分析 | 先追问关键细节，不跳过分析直接改 |
| 验证失败（编译/测试不通过） | 报告失败原因，回到步骤 3 重新分析 |
| 编码转换失败 | 记录异常文件路径，输出警告，不阻断修复流程 |
| 注释不合规 | 自动补齐后继续；无法自动补齐时标注待处理 |
| 用户在确认前追加新条件 | 回到步骤 3，重新分析新条件下的方案 |
| 修改后引入新问题 | 诚实告知，重新分析。不掩盖 |
| 方案无法二选一 | 追问用户优先级，不强行推荐 |
| 用户否决所有方案 | 追问具体不满意点，重新设计方案 |

## 禁止行为

- × 跳过分析直接改代码
- × 只给一种方案
- × 未确认就动手（除非用户已授权）
- × 改完不验证声称完成
- × 修改范围超出问题本身（顺手重构等）
- × 验证失败后掩盖错误
- × 新增/修改代码不写注释
- × 修改文件后不检查编码
